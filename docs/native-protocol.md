# Native messaging protocol

ScopeNest protocol version 1 uses Chrome's Native Messaging framing: a four-byte unsigned little-endian payload length followed by one UTF-8 JSON object. The maximum encoded message is 1 MiB. Standard output contains framed responses only.

## Request

```json
{
  "version": 1,
  "requestId": "a caller-generated correlation ID",
  "command": "launch_container",
  "data": {
    "id": "32 lowercase hexadecimal characters",
    "url": "https://example.com/"
  }
}
```

`requestId` must contain 1–128 characters. Every object is decoded strictly: unknown properties, incorrect JSON types, multiple values, and version mismatches are rejected.

## Response

```json
{
  "version": 1,
  "success": false,
  "requestId": "the matching request ID",
  "command": "launch_container",
  "error": { "message": "container is already running" },
  "errorCode": "ALREADY_RUNNING",
  "timestamp": "2026-07-13T16:00:00Z"
}
```

Successful responses use `data` instead of `error`/`errorCode`. Responses are never ambiguous plain text.

## Commands

| Command | Data | Result |
|---|---|---|
| `ping` | omitted or `{}` | host/protocol versions and startup-cleanup state |
| `get_status` | omitted or `{}` | versions, platform, data directory, counts, browsers, cleanup state, broken references, certificate resource issues, and trust capabilities |
| `list_containers` | omitted or `{}` | all reconciled container metadata |
| `get_running_containers` | omitted or `{}` | running subset |
| `create_container` | name, color, icon, browser type/executable | created container |
| `create_temporary_container` | same as create | created temporary container |
| `update_container` | ID plus editable create fields | updated container |
| `launch_container` | ID and optional HTTP(S) URL | launched container/process metadata, optional `proxyWarning`, and `directFallbackUsed` |
| `get_container_readiness` | ID | native authoritative readiness result including effective network, listener status, required certificate states, and warnings |
| `close_container` | ID | close requested for the Windows Job Object or Linux process group owned by this host instance |
| `delete_container` | ID | deletion result; running containers are rejected |
| `cleanup_temporary_containers` | omitted or `{}` | cleaned and pending ID lists |
| `validate_browser_path` | path | normalized valid path |
| `list_proxy_profiles` / `create_proxy_profile` / `update_proxy_profile` / `delete_proxy_profile` | profile data or ID | proxy profile metadata or deletion result |
| `test_proxy_listener` | loopback host and port | reachability, stable error code, and latency |
| `list_certificates` / `get_certificate` | omitted or certificate ID | certificate metadata |
| `import_certificate` | display name, Base64 content, decoded byte count | imported single X.509 CA certificate metadata |
| `install_certificate_trust` / `remove_certificate_trust` | certificate ID | updated Windows CurrentUser\\Root trust metadata |
| `acknowledge_manual_certificate_trust` | certificate ID, exact SHA-256 fingerprint, `platform: "linux"` | `manual_trust_acknowledged_unverified` certificate metadata with acknowledgment timestamp |
| `delete_certificate` | certificate ID | deletion result when unreferenced |

Container metadata includes a lifecycle `state` of `stopped`, `launching`, or `running`. `launching` is a persisted cross-process reservation. Its internal token is generated by the host and must match before a launch can be committed or rolled back.

Container network modes are explicit:

```text
direct
  no proxy or template references are accepted

template
  requires an environment template and inherits that template's proxy and certificate requirements

proxy
  requires a container proxy profile and overrides any template proxy while still allowing template certificate requirements
```

The native host resolves the effective proxy and required certificates while holding the metadata lock as part of launch reservation. Missing templates return `ENVIRONMENT_TEMPLATE_NOT_FOUND`, missing proxies return `PROXY_PROFILE_NOT_FOUND`, disabled proxies return `PROXY_PROFILE_DISABLED`, and missing required certificates return `CERTIFICATE_NOT_FOUND`. The host does not silently switch to direct networking for broken references.

Proxy profile data includes `enabled`, `protocol`, `host`, `port`, `bypassRules`, `certificateIds`, `healthCheck.enabled`, `healthCheck.timeoutMs`, and `unavailableBehavior`. Hosts are limited to `127.0.0.0/8`, `::1`, or `localhost`, and `localhost` is normalized to a loopback literal before persistence. IPv6 listener addresses are rendered with `net.JoinHostPort`. HTTP proxy transport uses `http=http://127.0.0.1:8080;https=http://127.0.0.1:8080`; HTTPS proxy transport uses `http=https://127.0.0.1:8443;https=https://127.0.0.1:8443`; SOCKS uses forms such as `--proxy-server=socks5://[::1]:1080`.

For proxy launches with `healthCheck.enabled`, the native host performs the authoritative listener check immediately before process startup. `warn` launches with proxy arguments and a warning without direct fallback; `block` returns `PROXY_LISTENER_UNAVAILABLE` without starting a browser; `direct` explicitly launches without proxy arguments or `--disable-quic` and sets `directFallbackUsed`. When health checks are disabled, ScopeNest skips reachability checks, launches with the proxy, and does not apply unavailable-listener behavior.

Linux manual trust acknowledgment is identity-bound metadata, not trust verification. The certificate ID, exact normalized-DER SHA-256 fingerprint, platform, and timestamp are persisted; a fingerprint mismatch invalidates the acknowledgment and never produces `trusted` or `verified` state.

Windows certificate trust operations target only `CurrentUser\Root`. The host persists recoverable trust states (`installing`, `removing`, `trust_error`) with an operation token and expected fingerprint before changing the trust store, verifies the exact managed DER afterward, and reconciles incomplete operations on startup. `delete_certificate` returns `CERTIFICATE_TRUST_MUST_BE_REMOVED_FIRST` for ScopeNest-owned certificates and `CERTIFICATE_REFERENCE_IN_USE` while a proxy or template still references the certificate. For trusted certificates not installed by ScopeNest, deletion is library-only and returns `windowsTrustUnchanged: true`. Certificate deletion operations are protected by persisted tombstones so startup can restore a staged directory when metadata still exists or finish deletion when metadata was already removed.

Both health commands include `startupCleanup` metadata. Its `state` is `pending`, `running`, `completed`, or `failed`; timestamps are included once the operation starts and finishes, and failures expose only the stable `INTERNAL_ERROR` code. The host writes its first valid native-message response before scheduling startup cleanup, so profile deletion cannot delay the initial health check.

The stored PID is lifecycle metadata, not process authority or proof that an unowned container is still running. `close_container` succeeds only when the current native-host process still holds the in-memory managed process created for that launch. It never reopens or kills a process from a persisted PID. When no matching managed process exists, reconciliation, relaunch, and deletion use Chromium profile-lock markers so PID reuse cannot preserve false running state.

The extension client applies command-specific response deadlines: 15 seconds for read/validation commands, 30 seconds for create/update/launch/close commands, and 5 minutes for deletion and manual temporary cleanup. These are client deadlines rather than protocol guarantees; a native operation is not cancelled merely because its caller stops waiting.

Representative error codes include `INVALID_REQUEST`, `UNSUPPORTED_VERSION`, `INVALID_REQUEST_ID`, `UNKNOWN_COMMAND`, `INVALID_DATA`, `INVALID_CONTAINER_ID`, `INVALID_NAME`, `INVALID_COLOR`, `INVALID_ICON`, `INVALID_BROWSER`, `INVALID_BROWSER_PATH`, `INVALID_URL`, `NOT_FOUND`, `ALREADY_LAUNCHING`, `ALREADY_RUNNING`, `LAUNCH_RESERVATION_LOST`, `CONTAINER_BUSY`, `CONTAINER_RUNNING`, `PROFILE_IN_USE`, `PROCESS_NOT_OWNED`, `LAUNCH_FAILED`, `CLOSE_FAILED`, `DELETE_FAILED`, `MESSAGE_TOO_LARGE`, and `INTERNAL_ERROR`.

Changing a request or response shape requires a new protocol version or a backward-compatible optional field. Extension and native-host command allowlists must remain synchronized.
